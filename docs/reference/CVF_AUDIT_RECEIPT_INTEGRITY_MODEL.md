## Purpose

Define how CVF audit receipts and evidence artifacts can be verified as
intact and untampered after the fact, so evaluators, auditors, and agents
have a single reference for the integrity model behind public evidence.

## Scope

CVF governance evidence receipts produced at `/api/execute`, the Git-based
hash chain used as the primary integrity anchor, and the boundary against
external tamper claims that require additional controls.

## Source

EA Track C — Audit Receipt Integrity. Integrity properties below are derived
from CVF's existing receipt schema and Git's content-addressable storage,
not from new runtime behavior added by this document.

## Protocol

The integrity protocol is: every `/api/execute` call emits a structured
receipt; the receipt and every committed artifact are anchored by Git commit
SHA; verification commands below traverse that anchor without trusting any
single rendered file.

## Enforcement

Integrity is enforced by Git's hash chain at the repository layer. CVF does
not add a parallel signing layer in this model. Verification commands in the
sections below are the operator-facing enforcement surface.

## Related Artifacts

- `docs/evidence/` evidence files
- `docs/benchmark/runs/` QBS run artifacts
- `docs/reviews/` governance reviews
- `docs/baselines/` authorization packets

## 1. What This Document Covers

CVF produces **governance evidence receipts** at every `/api/execute` call: a
`GovernanceEvidenceReceipt` embedded in the API response, plus audit events in the
control-plane ledger. This document defines how those receipts can be verified as
**intact and untampered** after the fact.

Integrity is not a claim — it is a verifiable property. This model describes what
makes CVF audit artifacts trustworthy, what the known gaps are, and what operators
must do to close them.

---

## 2. Git as a Content-Addressable Hash Chain

CVF's primary integrity mechanism is **Git's object model**, which is a
cryptographic hash chain by construction:

- Every commit SHA is the SHA-1 (now transitioning to SHA-256) hash of its tree +
  parent + metadata. Changing any byte in any committed file changes the commit SHA
  and every subsequent commit SHA in the chain.
- Every file committed to `docs/benchmark/`, `docs/evidence/`, and `docs/reviews/`
  is immutable from the moment it is pushed to the remote. The commit SHA is the
  integrity proof.
- Verification: `git log --oneline docs/benchmark/qbs-1/` traces every modification
  with its commit SHA. `git show <SHA>:<filepath>` retrieves the exact bytes at that
  commit.

### What This Means for QBS Artifacts

All QBS run artifacts under `docs/benchmark/runs/` are committed once and never
mutated after their corresponding `Record QBS*` commit. The pre-registration tag
(e.g. `qbs/preregister/qbs1-powered-single-provider-20260512-alibaba-r10`) locks the
exact Git tree at the moment of pre-registration, before any scored artifact exists.

To verify an artifact has not been changed since its record commit:

```bash
# Find the commit that added the file
git log --oneline --follow -- docs/benchmark/runs/qbs1-powered-single-provider-20260512-alibaba-r10/scored-results.json

# Verify the file content matches that commit exactly
git show <commit_sha>:docs/benchmark/runs/qbs1-powered-single-provider-20260512-alibaba-r10/scored-results.json | sha256sum
sha256sum docs/benchmark/runs/qbs1-powered-single-provider-20260512-alibaba-r10/scored-results.json
# Both SHA-256 values must match
```

## 3. Evidence Manifest (SHA-256 Snapshot)

As an additional layer beyond Git, a **SHA-256 manifest** can be generated over all
evidence files. This manifest is useful when:

- Providing evidence to an auditor who does not have Git access
- Doing a quick offline integrity check without running `git show`
- Creating a point-in-time snapshot of the evidence corpus for a compliance report

The manifest lives at `docs/evidence/MANIFEST_SHA256.json` and is generated by
`scripts/generate_evidence_manifest.py`. It records:

```json
{
  "generated_at": "2026-05-13T...",
  "git_commit": "<HEAD SHA at generation time>",
  "files": {
    "docs/benchmark/qbs-1/r10-post-score-analysis-qbs41.json": "<sha256hex>",
    "docs/evidence/claim-boundaries.md": "<sha256hex>",
    ...
  }
}
```

Verification: `scripts/verify_evidence_manifest.py` re-hashes every file in the
manifest and reports any mismatch. A mismatch means either the file was modified
outside of a Git commit (in-place edit without commit) or the manifest is stale and
needs regeneration after a legitimate update.

**Important:** The manifest is a snapshot, not a continuous monitor. It must be
regenerated after every evidence-modifying commit. The authoritative source of truth
is Git; the manifest is supplementary.

## 4. Governance Evidence Receipt Chain

Each `/api/execute` call produces a `GovernanceEvidenceReceipt` with:

| Field | Role |
| --- | --- |
| `receiptId` | `rcpt-<envelopeId>` — unique per request |
| `envelopeId` | `env-<uuid>` — governance envelope ID |
| `policySnapshotId` | SHA of active policy at decision time |
| `evidenceMode` | `live` — confirms real governance was applied |
| `decision` | `ALLOW / BLOCK / CLARIFY / NEEDS_APPROVAL` |
| `generatedAt` | ISO8601 timestamp |

The receipt is returned to the caller in the API response. It is not stored
server-side by default (stateless architecture). Operators who require long-term
receipt retention should:

1. Log receipts to an append-only store (database, S3, etc.) in their application layer.
2. Use the `receiptId` as the primary key.
3. Archive the `policySnapshotId` to prove which policy version was active.

For the development use case (local-first), the control-plane ledger at
`scripts/` writes audit events to JSONL. These files are the server-side audit
trail and can be integrity-checked the same way as evidence files.

## 5. Known Limitations

| Gap | Severity | Status |
| --- | --- | --- |
| No GPG commit signing | Medium | Deferred — requires private key infrastructure |
| No append-only server-side receipt store | Medium | Deferred — local-first scope; operator responsibility |
| SHA-1 Git object model | Low | Git is transitioning to SHA-256; SHA-1 preimage attacks are computationally infeasible for integrity purposes |
| Manifest requires manual regeneration | Low | Operator must regenerate after each evidence-modifying commit |
| Receipt not persisted by default | Medium | Documented above; operators must implement retention in their application layer |

**GPG signing note:** Until commit signing is enabled, Git history integrity depends
on access control to the remote repository (`Controlled-Vibe-Framework-CVF`). A
malicious actor with push access could rewrite history using `git push --force`. The
mitigation is: (1) branch protection on `main` requiring PRs, (2) limiting push
access to trusted maintainers. These are infrastructure-level controls, not
CVF-level controls.

## 6. Operator Verification Checklist

For a compliance review or handover, an auditor can verify CVF evidence integrity
with the following steps:

```bash
# 1. Clone the public repository at the audit point
git clone https://github.com/Blackbird081/Controlled-Vibe-Framework-CVF.git

# 2. Verify the pre-registration tag exists and matches the claim
git show qbs/preregister/qbs1-powered-single-provider-20260512-alibaba-r10

# 3. Verify no QBS run artifact was modified after its record commit
git log --oneline --follow -- docs/benchmark/runs/

# 4. Run the evidence manifest verifier
python scripts/verify_evidence_manifest.py

# 5. Spot-check a specific artifact against its commit SHA
git log --oneline -- docs/benchmark/qbs-1/r10-post-score-analysis-qbs41.json
git show <sha>:docs/benchmark/qbs-1/r10-post-score-analysis-qbs41.json | sha256sum
sha256sum docs/benchmark/qbs-1/r10-post-score-analysis-qbs41.json
```

All steps above require no CVF-specific tooling — only `git`, `python`, and
`sha256sum` (or `certutil -hashfile` on Windows).

## 7. Claim Boundary

- CVF **does not claim** GPG-signed commits or hardware-attested integrity.
- CVF **does claim** Git content-addressable immutability for all committed artifacts.
- The integrity model applies to the **public-sync repository** only. The provenance
  repository is private and push-disabled; its integrity is the same model but the
  remote is not publicly auditable.
- Receipt chain integrity is a **framework property**, not a claim about the AI
  output quality contained in those receipts.

## Claim Boundary

This model claims only that CVF artifacts are integrity-anchored by Git's
hash chain and that receipts emitted at `/api/execute` are structured for
post-hoc verification. It does not claim GPG signing, hardware attestation,
provenance-remote public auditability, or any property of the AI output
quality contained inside receipts.
