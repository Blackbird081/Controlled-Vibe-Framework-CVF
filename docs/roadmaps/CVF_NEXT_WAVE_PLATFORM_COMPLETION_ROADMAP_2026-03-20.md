## 1. Why This Roadmap Exists

The active system-unification wave is already:

- `COMPLETE FOR ACTIVE WAVE`
- `MATERIALLY DELIVERED`
- intentionally `DEPTH-FROZEN`

This file does **not** reopen that roadmap.

It exists to answer a different question:

**If CVF chooses to continue maturing after the current closure checkpoint, what is the most sensible next wave?**

## 2. Starting Point

Current independent readout from the new checkpoint:

- whole-system score: `8.2/10`
- active-path posture: `SUBSTANTIALLY ALIGNED`
- strongest safe claim: governance-first controlled execution system on the active reference path

What is already strong:

- canonical control model on the active reference path
- executable ownership for critical controls
- coder-facing governed reference loop

- strong auditability, baseline discipline, and stop-boundary governance

What still limits the strongest possible claim:

1. ecosystem breadth beyond the active reference path
2. uneven adapter maturity across extension families
3. reference-path strength still exceeds ecosystem-wide parity
4. safe positioning still stops short of "fully unified controlled autonomy everywhere"

## 3. Authorization Boundary

This roadmap is planning-only until one of the following happens:

1. a fresh independent reassessment identifies a new material gap that justifies reopening work
2. a new `GC-018` continuation candidate scores high enough to authorize the next wave

Until then, this roadmap must be read as:

- `prepared`
- `reviewable`
- `not yet active`

## 4. Next-Wave Objective

Move CVF from:

- substantially aligned on the active reference path

to:

- broader ecosystem parity with the same governed execution posture across a larger share of extension families and execution channels

The next wave is therefore **not** about redoing canonical unification.

It is about:

- ecosystem completion
- adapter parity
- broader control-loop realism
- claim-readiness hardening

## 5. Proposed Workstreams

### Workstream A — Ecosystem Breadth Parity

Goal:

- extend governed execution maturity beyond the current active reference path

Focus:

- identify extension families still relying on reference-runtime assumptions
- align their entrypoints to canonical phases, approval boundaries, and evidence expectations
- remove remaining pockets of uneven control maturity

Definition of done:

- additional extension families can demonstrate governed execution without falling back to legacy semantics or manual interpretation

### Workstream B — Adapter Maturity And Execution Surface Parity

Goal:

- reduce the gap between reference-path execution and ecosystem-wide adapter realism

Focus:

- production-grade adapter bindings
- broader handler coverage
- consistent step-result propagation and freeze evidence outside the current main path

Definition of done:

- governed workflow realism is no longer strongest only in the current reference path

### Workstream C — Cross-Channel Claim Readiness

Goal:

- raise the strongest safe product claim without overstating current maturity

Focus:

- verify where coder-facing and non-coder governed paths are equivalent
- tighten evidence for cross-channel parity
- only upgrade claims once parity is demonstrated, not assumed

Definition of done:

- positioning can move beyond "substantially aligned on the active reference path" with evidence, not optimism

### Workstream D — Ecosystem Control-Coverage Audit

Goal:

- make governance coverage auditable across more than the current core surfaces

Focus:

- expand control-matrix evidence into secondary extension families
- confirm which rules are runtime-owned, gateway-owned, approval-owned, or CI-owned across more of the ecosystem
- identify residual control gaps before they become claim drift

Definition of done:

- future reassessment can score ecosystem maturity from evidence instead of scattered inspection

## 6. Priority Order

Recommended order for the next wave if it is authorized:

1. Workstream A — Ecosystem breadth parity
2. Workstream B — Adapter maturity and execution parity
3. Workstream D — Ecosystem control-coverage audit
4. Workstream C — Cross-channel claim readiness

Reasoning:

- parity and execution realism should improve before claims are upgraded
- audit coverage should expand before stronger positioning statements are made

## 7. Suggested Delivery Phases

### Phase N1 — Discovery And Scoping

- inventory extension families outside the active path
- classify them by governance maturity
- find the top 3 highest-value parity gaps

### Phase N2 — Execution Parity

- implement the highest-value adapter and workflow parity fixes
- add evidence-backed governed execution receipts in those families

### Phase N3 — Coverage And Claims

- extend control-matrix evidence
- rerun independent reassessment
- decide whether stronger platform-level claims are now justified

## 8. Success Metrics

This next wave should be considered successful only if it produces at least 4 measurable outcomes:

1. at least `3` additional non-reference execution surfaces reach the same governed control posture as the active path
2. adapter-bound governed execution evidence exists outside the current strongest runtime line
3. control-matrix coverage expands beyond the current core families with reviewable proof
4. a future independent reassessment can justify a stronger claim than today's active-path-limited posture

## 9. Stop Rules

Even if this next wave is authorized, it should stop when:

- parity gains become mostly repetitive proof volume
- new batches add more narrative than executable closure
- claim-readiness no longer improves materially
- another newer material gap outranks ecosystem parity
