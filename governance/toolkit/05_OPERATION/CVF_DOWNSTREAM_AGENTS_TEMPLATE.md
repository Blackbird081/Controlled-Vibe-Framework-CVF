## Mandatory Governance Proof

Any test, roadmap closure, release gate, demo proof, or public claim that asserts
CVF governance behavior — risk classification, approval flow, phase gates, DLP
filtering, bypass detection, output validation, provider routing, audit trail
updates, or CVF controlling AI/agent behavior — **must use a real provider API call**.

Mock mode is allowed **only** for pure UI structure checks (navigation, routing,
static badges, layout, RBAC pages) that do not assert AI governance behavior.

## Phase Model

This project uses the CVF phase model:

```
INTAKE → DESIGN → BUILD → REVIEW → FREEZE
```

| Phase   | Purpose                                    | Allowed actions                         |
|---------|--------------------------------------------|-----------------------------------------|
| INTAKE  | Capture intent, scope, risk classification | Read, classify, ask clarifying questions |
| DESIGN  | Produce plans, architecture, specs         | Draft artifacts, get approval           |
| BUILD   | Implement planned work                     | Code, test, integrate                   |
| REVIEW  | Validate against spec + governance         | Review, audit, regression test          |
| FREEZE  | Lock artifacts, prepare release            | Tag, document, close tranche            |

Do NOT skip phases. Do NOT assume a phase without reading the manifest.

## Live Governance Evidence Rule

For any claim that CVF governs AI/agent behavior in this project:

- You MUST invoke a real provider API call.
- You MUST record the request/response in the evidence artifact.
- You MUST NOT use mocked provider output as governance proof.

Mock providers are allowed only for:
- UI rendering tests (layout, navigation, static badges)
- Component-level tests that do not assert governance decisions

## Workspace Isolation Rule

This project must remain isolated from the CVF core repository:

- Do NOT run downstream project tasks inside the CVF core directory.
- Do NOT commit downstream project artifacts into the CVF core repository.
- CVF core path (`{{CVF_CORE_PATH}}`) is **read-only reference** for this project.
- Your IDE/terminal working directory must always be this project root.

## Risk Classification

Before acting on any request, classify its risk level:

| Level | Meaning                                       |
|-------|-----------------------------------------------|
| R0    | No production impact, no governance surface   |
| R1    | Low risk, standard workflow, no approval needed |
| R2    | Elevated risk — requires human review         |
| R3    | Critical — requires formal approval before action |

Requests involving production data, external APIs, security changes, or
governance-relevant behavior are at minimum R2.

## Handoff and Tranche Closure Protocol

When closing a tranche or handing off to another agent:

1. Update or create a handoff record in `docs/` (e.g. `AGENT_HANDOFF.md`).
2. Record:
   - Tranche id, status (CLOSED / IN_PROGRESS)
   - What was done and what was not done
   - Active risk and open decisions
   - Next governed move
3. Commit all artifacts before declaring closure.
4. Do NOT declare a tranche closed if tests are failing or artifacts are missing.

Governance has priority over speed, user preference, and agent autonomy.
